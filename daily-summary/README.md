# Daily Summary Folder

**Purpose:** Static, git-committed AI-driven summaries from automated "Routine Run" jobs. Free GitHub-only setup (no S3, Slack, or paid infrastructure).

## What goes in this folder

Two file families, both auto-generated by GitHub Actions on a daily cadence:

| Filename pattern | Producer | Purpose |
|---|---|---|
| `<YYYY-MM-DD>.md` | `DailyFailureAnalyzer.java` | Daily test-failure executive summary (top failures, root causes, trend, action items) |
| `security-<YYYY-MM-DD>.md` | `APISecurityScanner.java` | Daily security-test triage (OWASP-tagged failures, AppSec-friendly format) |

Files are **append-only history** — past summaries persist in git so trends are visible across weeks/months.

## How files are generated

### Workflow: `.github/workflows/daily-ai-analysis.yml`

Triggers:

1. **Cron daily at 04:00 UTC** (after most CI runs settle) — `cron: '0 4 * * *'`
2. **After any Parallel Suite run completes** (chained via `workflow_run`)
3. **Manual `workflow_dispatch`** — for ad-hoc runs

What it does:

```text
GitHub Actions
   ↓
Checkout repo (gives access to test-output/bug-detection-report.json)
   ↓
mvn exec:java -Dexec.mainClass=DailyFailureAnalyzer
   ↓
mvn exec:java -Dexec.mainClass=APISecurityScanner
   ↓
Commit daily-summary/*.md back to main
   ↓
Done — summaries available at raw GitHub URL
```

The commit-back step uses a workflow-scoped GITHUB_TOKEN — no PAT setup needed. Free.

### Optional Claude API enrichment

If `CLAUDE_API_KEY` is set as a GitHub Actions secret, the summaries will include AI-narrated:

- Executive summary (4-6 sentences, plain English, manager-readable)
- Per-finding root cause + suggested fix
- Security triage with impact + next-step recommendation

**Without** the secret, the summaries fall back to deterministic rules-based analysis. They're shorter but still useful — counts, classifications, ranked failures, trend vs yesterday, action items.

## Reading the summaries

### As a developer (your daily standup)

Open the most recent `<YYYY-MM-DD>.md` file. The structure:

1. **Executive Summary** (1 paragraph) — what changed since yesterday
2. **Top Failures (Ranked)** — by REAL_BUG > LOCATOR_CHANGE > FLAKY_TEST > ENVIRONMENT_ISSUE, then by confidence
3. **AI Executive Summary** — Claude's prose, when API key set
4. **Action Items for Next CI Run** — 3-5 concrete next steps
5. **Ready-Bug Files Generated (last 24h)** — links to per-failure markdown for Jira filing

### As an AppSec engineer (weekly review)

Open `security-<YYYY-MM-DD>.md`. The structure:

1. **Security Posture** — GREEN if no OWASP-tagged failures; otherwise count
2. **OWASP Category Breakdown** — failures grouped by A01 / A05 / A06 / etc.
3. **AI Security Triage** — Claude's most-important-finding-to-investigate
4. **Recommended Actions** — concrete next steps per category

## What this folder is NOT for

- ❌ Long-form bug reports — those go in `ready-bug/` (one .md per failure, Jira-pasteable)
- ❌ Test execution reports — those are at `reports/detail-report/` and `reports/client-report/`
- ❌ Code changes — those go in source files

## Reading from GitHub (raw URL pattern)

Once committed, every summary file is reachable at:

```
https://raw.githubusercontent.com/eg-abhiyantsingh/Egalvanic_Web-main/main/daily-summary/<YYYY-MM-DD>.md
```

This is the "free S3 alternative" the user asked for. Any external tool (Claude UI, Slack incoming-webhook, Notion sync) can fetch these directly — no auth, no credentials, no paid infra.

## Cleanup policy

- Keep the **last 90 days** of summaries in git
- Older summaries: optional manual archive to `daily-summary/archive/<YYYY>/<MM>/`
- The workflow does NOT auto-prune — append-only history is the value

## Customizing

| What to change | Where |
|---|---|
| Cron schedule | `.github/workflows/daily-ai-analysis.yml` line `cron:` |
| Failure ranking | `DailyFailureAnalyzer.classificationPriority()` |
| Security test class allowlist | `APISecurityScanner.SECURITY_CLASSES` |
| AI threshold (confidence at which to skip Claude) | `AIBugAnalyzer.analyze()` (line `confidence >= 80`) |
| Trend window (days compared) | `DailyFailureAnalyzer.INPUT_PREV` (currently yesterday only) |

## Privacy / what gets sent to Claude

Same redaction layer as `AIBugAnalyzer` and `ReadyBugGenerator`:

- Bearer tokens stripped
- JWTs (`eyJ...` patterns) stripped
- `password=` and `token=` values stripped

If your environment has stricter rules: don't set `CLAUDE_API_KEY` and the framework runs rules-only.
