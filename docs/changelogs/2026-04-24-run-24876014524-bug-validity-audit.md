## In-depth explanation (for learning + manager reporting)

### Why "inverted polarity" is the most dangerous failure mode
A test like `testBUG007_DuplicateAPICalls` asserts that duplicate API calls still happen. When the underlying bug is FIXED, duplicate calls stop — and the test goes RED. To a casual reader, "red test" reads as "something broke". For a client report, 2 red tests in BugHunt makes the product look worse than it is.

The fix is simple (flip the assertion from `assertTrue(bugPresent)` to `assertFalse(bugPresent)` and update the message) but finding them across a 50+ file test suite requires the "Bug may be fixed" text grep. I did that grep above and found 6 files with the pattern.

### Why backend-indexing retries are a cross-cutting pattern
Any test that creates a record and then searches for it is vulnerable. The backend's write-to-search-index latency can be 5-15s in production-like environments. Tests that search immediately after create hit the gap. Three tests in this suite have the exact same symptom (WO create + Asset create I fixed before). Same fix: retry `isXVisible` up to 30s with 3s intervals.

### Why stale-element errors cluster in Edit flows
MUI's Edit drawers re-render the form as the user types. If a test caches a `WebElement` reference across the re-render, the reference goes stale. The fix is to re-find the element via its locator each time — don't cache `WebElement` variables across user actions.

### Why "4 real bugs out of 16 failures" is actually a healthy signal
Any sufficiently-large test suite will have ~70-80% of failures be test-side issues (timing, selectors, env). If 100% of failures were product bugs, the suite would be over-sensitive (flaky). If 0% were product bugs, the suite would be useless. A 25% real-bug rate means the suite is doing its job — surfacing regressions while also catching its own noise, which you can then triage.
